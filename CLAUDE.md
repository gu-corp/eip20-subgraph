# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Graph Protocol subgraph that indexes ERC20 `Transfer` events and maintains derived balances/total-supply per token and account. Mappings are AssemblyScript (compiled to WASM); the data model is in `schema.graphql`.

## Commands

```bash
npm install
npm run codegen        # generate types from subgraph.yaml + ABIs + schema -> generated/
npm run build          # compile mappings to WASM -> build/
npm run clean          # rimraf build/ generated/
```

There is no test suite and no linter configured — `codegen` + `build` are the only local verification. A build failure is the signal that mappings/manifest are wrong.

**codegen/build always operate on `subgraph.yaml`**, which is a throwaway file. Never edit `subgraph.yaml` as the source of truth — it is overwritten per network. To build/test a given network locally, copy its manifest first:

```bash
cp subgraph-sandbox1.yaml subgraph.yaml && npm run codegen && npm run build
```

## Architecture

### Per-network manifests (not `networks.json`)

Each target network has its own committed manifest: `subgraph-joc.yaml`, `subgraph-joc-testnet.yaml`, `subgraph-sepolia.yaml`, `subgraph-mainnet.yaml`, `subgraph-sandbox1.yaml`. The deploy flow is always `cp subgraph-<net>.yaml subgraph.yaml` → codegen → build → deploy. `networks.json` exists but the workflows do not use it; the per-network yaml is what matters.

Two manifest shapes are in use:
- **Address-less single data source** (joc, sandbox1): one `ethereum/contract` data source named `EIP20` with no `source.address`, so it indexes `Transfer` events from *every* contract on the chain.
- **Per-token data sources** (mainnet): many data sources, one per token address, generated by `scripts/gen-subgraph-yaml.js`.

`scripts/gen-subgraph-yaml.js --network <slug>` builds a per-token manifest by filtering a remote token list by chainId and resolving each token's contract-creation block via a Blockscout API. Add new networks to its `NETWORK_CONFIG`.

### The data source MUST be named `EIP20`

`src/mapping.ts` imports generated types from `../generated/EIP20/IERC20`. codegen writes those types to `generated/<data-source-name>/`, so **at least one data source (or template) must be named `EIP20`** or the build fails with `File 'generated/EIP20/IERC20.ts' not found`. Renaming the data source requires updating the imports in `src/mapping.ts`.

### Mapping logic (`src/mapping.ts`)

Single handler `handleTransfer`. Key behaviors:
- `fetchERC20` lazily creates an `ERC20Contract` on first sight and reads `name/symbol/decimals` via `try_*` calls; missing decimals default to 18, absurd values are capped at 36 (`normalizeDecimals`).
- Balances are keyed `contractHex/accountHex`, with the special id `contractHex/totalSupply` for supply.
- Mints (`from == 0x0`) increase total supply; burns (`to == 0x0`) decrease it; normal transfers move exact balances. `value` is the decimal-scaled amount, `valueExact` the raw integer.

`schema.graphql`: `ERC20Contract`, `ERC20Transfer`, and `Transaction` are `immutable`; `Account` and `ERC20Balance` are mutable. `ERC20Transfer` implements the `Event` interface. Most reverse relations use `@derivedFrom`.

## Deployment (GitHub Actions, `workflow_dispatch`)

All deploys are manual workflows in `.github/workflows/`. Two distinct targets:

- **The Graph Studio** (`deploy-eip20-{joc,joc-testnet,sepolia,mainnet}.yml`): `graph auth <DEPLOY_KEY>` then `graph deploy --node https://api.studio.thegraph.com/deploy/ --ipfs https://api.thegraph.com/ipfs/`. Deploy keys are per-network repo secrets.

- **Self-hosted sandbox1 graph-node** (`deploy-eip20-sandbox1.yml`): deploys to `admin.graphnode.sandbox1.japanopenchain.org` (behind Caddy Basic Auth), subgraph name `gugroup/eip-20`. Notable constraints:
  - Requires `graph create` before first deploy (step runs with `continue-on-error: true` so redeploys don't fail).
  - `network:` in the manifest must be `sandbox1` to match the node's `ETHEREUM_NETWORK`.
  - **IPFS upload cannot use credentials in the URL.** Modern graph-cli uploads via `fetch()`, which rejects `user:pass@host` URLs. So `scripts/ipfs-auth-proxy.mjs` runs locally, injects the `Authorization: Basic` header, and `--ipfs` points at `http://127.0.0.1:5001`. The admin RPC (`--node`) still accepts credentials inline. Password comes from the `SANDBOX1_GRAPHNODE_ADMIN_PASSWORD` secret.
  - Query endpoint: `https://app.graphnode.sandbox1.japanopenchain.org/subgraphs/name/gugroup/eip-20`.

## Working agreement

- **Do not `git commit` or `git push` automatically.** Make the file changes and stop; commit/push only when the user explicitly asks in that request.

## Conventions

- `src/mapping.ts` uses tabs for indentation. This is AssemblyScript, not TypeScript — no closures/generics/standard-lib assumptions; use `@graphprotocol/graph-ts` and `@amxx/graphprotocol-utils` helpers.
- `generated/` and `build/` are gitignored; do not commit them.
